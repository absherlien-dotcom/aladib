# سياسة حماية شبكة الأديب نت

## حقيقة مهمة

واجهة الموقع لا تقرر أن رقم الكرت صحيح أو خاطئ. أي تحقق داخل HTML أو JavaScript يمكن للمستخدم تعديله أو تجاوزه، لذلك يجب أن يكون القرار النهائي داخل MikroTik أو RADIUS أو خادم مصادقة خاص لا يمكن الوصول إليه مباشرة من الإنترنت.

المسار الآمن المقترح:

```text
جهاز العميل
   ↓ HTTPS أو HTTP-CHAP
بوابة MikroTik Hotspot
   ↓ شبكة إدارة خاصة
RADIUS / نظام الكروت
   ↓ اتصال داخلي فقط
قاعدة البيانات
```

## ما يجب ألا يوجد داخل الموقع

- كلمات مرور MikroTik أو RADIUS.
- مفاتيح API أو بيانات قاعدة البيانات.
- قائمة أرقام الكروت الصالحة.
- منطق يمنح الاتصال اعتمادًا على JavaScript فقط.
- رابط مباشر لقاعدة البيانات أو منفذ قاعدة بيانات مفتوح للإنترنت.

## حماية التحقق من أرقام الكروت

1. أنشئ الأكواد عشوائيًا باستخدام مولد تشفيري، ولا تستخدم أرقامًا متسلسلة يمكن تخمينها.
2. اجعل كل كرت ذا حالة واضحة: جديد، مستخدم، منتهي، موقوف.
3. تحقق من الكرت في MikroTik/RADIUS فقط، وليس في المتصفح.
4. ضع حدًا لمحاولات الدخول بحسب عنوان IP وعنوان MAC ورقم الكرت.
5. بعد عدة محاولات فاشلة، طبّق انتظارًا مؤقتًا وسجّل الحدث للمراجعة.
6. لا تعرض في رسالة الخطأ هل الرقم موجود أو غير موجود؛ استخدم رسالة عامة مثل: "تعذر تسجيل الدخول، تحقق من بيانات الكرت".
7. لا تسجل رقم الكرت كاملًا في السجلات. أخفِ معظم أرقامه عند الحاجة للمراجعة.
8. امنع استخدام الكرت بعد انتهاء مدته أو استهلاك رصيده، وطبّق ذلك من الخادم.
9. إن كان النظام يسمح، اربط الكرت بأول جهاز أو ضع سياسة تمنع الاستخدام المتزامن.

## حماية قاعدة البيانات

- قاعدة البيانات تكون على عنوان خاص وغير قابلة للوصول مباشرة من الإنترنت.
- اسمح بالاتصال بها فقط من خادم المصادقة المحدد وجدار ناري محدد.
- استخدم حساب قاعدة بيانات بصلاحيات أقل قدر ممكن.
- لا تستخدم حساب المدير العام للتطبيق.
- استخدم استعلامات مهيأة Parameterized Queries ولا تبنِ الاستعلامات من نص المستخدم.
- شفّر النسخ الاحتياطية واحفظها خارج الخادم الأساسي.
- غيّر كلمات المرور والأسرار دوريًا، ولا ترسلها في المحادثات أو البريد العادي.
- لا تضع الأسرار في GitHub؛ استخدم متغيرات بيئة مشفرة في منصة الاستضافة.

## حماية MikroTik

- استخدم `http-chap` أو `https`، ولا تستخدم `http-pap` عبر شبكة غير مشفرة.
- حدّث RouterOS إلى إصدار مدعوم ومستقر.
- عطّل Telnet وFTP وواجهة WWW الإدارية إن لم تكن ضرورية.
- اسمح بـ Winbox وSSH فقط من شبكة إدارة داخلية أو VPN وعناوين محددة.
- لا تسمح بإدارة الراوتر من شبكة العملاء.
- استخدم كلمة مرور إدارة طويلة وفريدة، وحسابًا منفصلًا لكل مسؤول.
- اجعل RADIUS shared secret عشوائيًا وطويلًا ولا تحفظه في ملفات الموقع.
- افصل شبكة العملاء عن شبكة الإدارة وقاعدة البيانات باستخدام VLAN وجدار ناري.
- راقب محاولات الدخول الفاشلة والتنبيهات وتغييرات الإعدادات.
- خذ نسخة احتياطية مشفرة بعد كل تغيير مهم، واختبر استعادتها.

## حماية GitHub وVercel

- فعّل المصادقة الثنائية لحساب GitHub وحساب Vercel.
- لا تمنح صلاحية الكتابة للمستودع إلا للأشخاص الضروريين.
- فعّل حماية فرع `main` والمراجعة قبل الدمج عندما يشارك أكثر من مطور.
- اجعل أي مستودع يحتوي خادمًا أو إعدادات داخلية مستودعًا خاصًا.
- راجع سجل النشر والتغييرات بانتظام.

## الطبقات المطبقة في نسخة Vercel

- Content Security Policy لمنع تحميل سكربتات ومصادر غير مصرح بها.
- منع تضمين الموقع داخل iframe لحماية النقر الخادع.
- HTTPS Strict Transport Security.
- منع تخمين نوع الملفات `nosniff`.
- منع إرسال معلومات الإحالة.
- تعطيل صلاحيات الكاميرا والميكروفون والموقع والأجهزة غير المطلوبة.
- منع تخزين صفحة تسجيل الدخول في ذاكرة المتصفح المؤقتة.
- منع رفع ملفات الأسرار الشائعة عبر `.gitignore`.

## حدود الحماية

هذه الإجراءات تقلل المخاطر بقوة، لكنها لا تجعل أي نظام محصنًا 100%. الحماية المستمرة تحتاج تحديثات، مراجعة إعدادات الراوتر، فحص السجلات، واختبارات أمنية دورية من مختص مصرح له.
